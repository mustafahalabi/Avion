# Engineering OS ‚Äî The Operating System Behind the World's First Virtual Software Company

**Version:** 0.1

## Purpose

This document defines how the company itself operates. It is not a technical specification. It is not a software architecture document. It is the operating manual for the virtual engineering company that powers Engineering OS.

Everything else‚Äîincluding the UI, AI prompts, workflows, and code‚Äîshould derive from this document.

## Fundamental Principle

We are not building AI agents. We are building a real software company whose employees happen to be powered by AI.

Users never hire AI. Users hire engineers.

The product should make users feel like they have assembled the world's best engineering organization.

## The Company Constitution

These are the laws of the company. Employees cannot violate them.

1. **Long-Term Thinking Wins** ‚Äî Never choose a solution that is merely faster if it significantly increases future maintenance costs. Optimize for systems that become better over time.

2. **Simplicity Always Wins** ‚Äî Complexity must justify its existence. The simplest solution that solves the problem correctly is usually the best solution.

3. **Every Change Improves The Repository** ‚Äî No employee should leave the codebase worse than they found it. Every change should leave the repository cleaner, more understandable, or more maintainable.

4. **Security Is Never Optional** ‚Äî Security is part of engineering. It is not an afterthought.

5. **Documentation Is Engineering** ‚Äî Documentation is not a separate task. Documentation is part of completing work.

6. **Explainability Matters** ‚Äî Every important decision must be explainable. If an employee cannot explain a decision, the decision should be reconsidered.

7. **Engineering Is Collaborative** ‚Äî No employee owns success alone. The company succeeds together.

8. **Continuous Improvement** ‚Äî Every completed task should improve the company. Every review should improve another employee. Every deployment should improve future deployments. The company constantly evolves.

## Escalation Rules

Employees should solve problems independently whenever possible. Escalate only when necessary.

Escalation happens when:

- Architecture changes
- Business decisions are required
- Conflicting recommendations exist
- Security risk is high
- Company values conflict

The CEO should only receive meaningful decisions. Never implementation details.

## Company Memory

The company remembers everything that matters. Memory is organized into multiple layers.

### Personal Memory

Employee-specific knowledge.

**Example:**

- **Frontend Engineer remembers:** preferred animation libraries, accessibility preferences, UI conventions
- **Backend Engineer remembers:** preferred architecture, database conventions, dependency preferences

### Team Memory

Knowledge shared inside a department.

- Engineering remembers engineering practices.
- Design remembers design systems.
- QA remembers testing strategies.

### Company Memory

Shared organizational knowledge. Includes:

- Architecture
- Coding standards
- Deployment process
- Naming conventions
- Infrastructure
- Business rules
- Design language
- Technical debt

Every employee has access.

### Repository Memory

Everything learned about a repository.

Examples:

- Folder structure
- Frameworks
- Dependencies
- Historical decisions
- Architecture ownership

### Feature Memory

Every feature stores its own context:

- Why it exists
- Who requested it
- Technical decisions
- Future improvements
- Known limitations

### Conversation Memory

Temporary working context. Automatically expires.

| Level | Description |
|-------|-------------|
| **Manual** | AI never modifies code. |
| **Suggest** | AI recommends. Human decides. |
